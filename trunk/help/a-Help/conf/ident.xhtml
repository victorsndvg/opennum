A distributor can use a particular word as package name, use his own background images and logo. This name will also be the name for the configuration folder and the materials data base.

The package name

The main file that opens the graphical interface is the Pyhton script <installdir>/code/opennum.py, where <installdir> is the installation folder. To use your own word as package name, just rename this file, maintaining the .py extension. The new name will appear in the title bar. In case you want to distribute several packages within the same installation, you can make several copies of opennum.py, a rename every copy with the appropriate name.

For example, if you want to distribute a package for solving thermal problems, just rename opennum.py to thermal.py if you want that "thermal" be the package name.

Changing the associated images.

OpenNum uses two image files in the graphical interface (here <installdir> is the installation folder):

  • <installdir>/images/background.png is the background image that appears in the graphic scene when OpenNum starts.
  • <installdir>/images/icon.xpm is the icon image that appears in the Task bar in those operating systems that allow it.

You can change those images by copying in the previous files your preferred ones, maintaing the original formats (PNG and XPM).

In case you want to distribute several packages within the same installation, you can make several copies of these files and rename them as background.<name>.png and <name>.xpm, where <name> is every package name. ¿No debería ser icon.name.xpm?.

Renaming the configuration folder

With the first execution of the main file, OpenNum creates a configuration folder in the user's HOME. The name of this folder is the package name.

Since there is no guarantee that the user has write permission on the installation folder, this folder is used to save:

  • the name of the last working folder in the file lastfolder.dat;
  • a modifiable copy of the materials database in the subfolder materialsDB;
  • a copy of the executable files in the subfolder solvers; this is useful when the user wants to use his own version of the executable files or to launch a remote run.
  • a temporal subfolder with an execution when launching a remote run.

These are the usual locations of the configuration folder in the user's HOME, depending on the operating system:

  • GNU/Linux: /home/<user>/.<name>.

  • Windows XP: C:\Documents and Settings\<user>\Application Data\<name>.

  • Windows 7: C:\Users\<user>\AppData\Roaming\<name>.

  • Mac OS X: /Users/<user>/Library/Application Support/<name>.

Here <user> stands for the username and <name> for the package name.

Renaming the materials database

OpenNum brings an example of materials database named materials.xml in the folder <installdir>/materialsDB, where <installdir> is the installation folder. You can modifiy this file for your own purposes. In case you want to distribute several packages within the same installation, you can make several copies of the materials database and rename it as materials.<name>.xml where <name> is every package name.

With the first execution of the graphical interface, OpenNum creates a configuration folder in the user's HOME and saves there a modifiable copy of the materials database in the subfolder materialsDB. The user could modify this copy through the graphical interface.

Every time an executable file is called from the graphical interface, a copy of the latter materials database is saved in the working folder with the name materials.dat.xml. Thus, the executable file can rely on that a copy of the lastest modified materials database exists in the working folder.
