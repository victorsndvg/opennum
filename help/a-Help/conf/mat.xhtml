Materials database

On this page:

Description

A package can be distributed with a materials database, common for all the applications. A materials database can store the physical properties of a set of materials. That is useful when working with scientific solvers that simulate a physical phenomenon: the thermal behavior of a motor, the elastic deformation of a bridge, etc. In those cases, several physical properties must be defined and associated with each part of the domain. The materials database permits to do such association in two steps:

  1. the physical properties are defined for several materials in the materials database.
  2. a material is associated with each part of the domain using a specific process designed by the distributor (for example, allowing the user to choose the materials in a menu item).

Folder location

The materials database distributed with the package must be located on <installdir>/materialsDB, where <installdir> is the installation folder. As it was commented in the section Renaming the materials database the name of the materials database can be materials.xml or materials.<name>.xml where <name> is every package name.

With the first execution of the graphical interface, OpenNum creates a configuration folder in the user's HOME and saves there a modifiable copy of the materials database in the subfolder materialsDB. The user could modify this copy through the graphical interface.

Every time an executable file is called from the graphical interface, a copy of the latter materials database is saved in the working folder with the name materials.dat.xml. Thus, the executable file can rely on that a copy of the latest modified materials database exists in the working folder.

The materials database file

The materials database file is a XML file encoded in utf-8 and constructed in the same way than the menu file. The materials.xml file is a tree composed of menu items, submenu items, structures and leafs. It describes the tree shown in the menu item Materials database of the graphical interface.

The distributor have freedom to order the materials in the database and to choose which physical properties can be defined for them. We recommend to save all materials at the same level in the hierarchy in order to ease the list of the existing materials in other XML structures of the menu. Solvers, or its associated data converters, must know how to read the materials database.

The complete list of elements and attributes allowed in the menu's description is in the Section XML syntax.

Some examples are discussed in the sample data attached to OpenNum.
